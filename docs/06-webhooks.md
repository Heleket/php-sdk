# 06 — Webhooks

Webhooks deliver payment and payout status updates to your server. **Always** verify the signature before trusting the payload.

## How Heleket signs webhooks

For every webhook event Heleket POSTs a JSON body containing your fields plus a `sign` field. The signature is computed exactly like outbound request signatures:

```
sign = md5( base64_encode(json_body_without_sign_field) . apiKey )
```

The key depends on the webhook type:

| Webhook type | Key to verify with |
|---|---|
| `payment`, `wallet` | Payment API key |
| `payout` | Payout API key |

If you mix them up, every webhook fails.

## Verifying a webhook

Use `Heleket\Webhook\WebhookVerifier`. Both entry points below verify by decoding + re-encoding the JSON, then checking the MD5 — pick whichever is more convenient for your handler.

### Verify from the raw request body

```php
use Heleket\Exception\SignatureException;
use Heleket\Webhook\WebhookVerifier;

$rawBody = file_get_contents('php://input');
$verifier = new WebhookVerifier($paymentApiKey);

try {
    $payload = $verifier->verifyRawBody($rawBody);
} catch (SignatureException $exception) {
    http_response_code(400);
    error_log('[heleket] bad signature: ' . $exception->getMessage());
    return;
}

// $payload is a verified WebhookPayload value object.
if ($payload->isSuccessful()) {
    markOrderPaid($payload->getOrderId(), $payload->getAmount());
}

http_response_code(200);
echo 'OK';
```

### Verify from an already-decoded array

```php
$payload = $verifier->verify($_POST);  // or any array decoded from JSON
```

## The slash-escaping trap

PHP's `json_encode()` escapes forward slashes by default (`https://x` → `https:\/\/x`). Most other languages do not. Heleket's server signs using PHP-style encoding.

**Practical consequence:** both `verify()` and `verifyRawBody()` decode the incoming JSON and re-encode it with `JSON_UNESCAPED_UNICODE` (which keeps slashes escaped) before hashing. If the incoming payload differs from what PHP would produce — for example, if it was generated by a JavaScript service that left slashes raw — verification fails.

**Mitigation:** when integrating with Heleket directly, this is a non-issue (Heleket uses PHP-style encoding). If you proxy a webhook through a non-PHP service that re-serializes the JSON, switch that service to escape forward slashes the same way PHP does (or stop re-serializing — pass the raw body through).

## Reading the verified payload

`WebhookPayload` exposes typed accessors for the common fields. Anything not listed is available via `toArray()`.

```php
$payload->getType();      // "payment", "wallet", or "payout"
$payload->isPayment();    // true for payment/wallet, false for payout
$payload->isPayout();
$payload->getUuid();
$payload->getOrderId();
$payload->getStatus();    // e.g. "paid", "wrong_amount", "process"
$payload->isFinal();      // true if the status will not change
$payload->isSuccessful(); // true for paid / paid_over
$payload->getAmount();
$payload->getTxid();
$payload->getNetwork();
$payload->toArray();      // full payload as decoded
```

`isFinal()` and `isSuccessful()` use the status enums from [10 — Reference](10-reference.md).

## Recommended handler shape

```php
http_response_code(200);

try {
    $payload = $verifier->verifyRawBody(file_get_contents('php://input'));
} catch (SignatureException $e) {
    http_response_code(400);
    return;
}

// Respond fast. Queue heavy work — Heleket retries until you 200.
queueOrderUpdate($payload->toArray());
```

## Idempotency and replay protection

Heleket may deliver the same status more than once. Common causes:

- **Retries** — if you respond non-2xx (or time out), Heleket retries with the same payload.
- **Manual replays** — operators (and your own code via `$client->resendWebhook(...)`) can re-send historical events.
- **Network drops** — your edge accepted the request but the response never reached Heleket, so it retries.

A signature-valid webhook captured off the wire can also be replayed by an attacker who proxies it back to your endpoint. The signature alone does **not** prove freshness.

**You must de-duplicate.** Treat each `(uuid, status)` pair as a unique event and store it before doing any side-effect work.

### Minimal idempotent handler

```php
use Heleket\Exception\SignatureException;
use Heleket\Webhook\WebhookVerifier;

$verifier = new WebhookVerifier($paymentApiKey);

try {
    $payload = $verifier->verifyRawBody(file_get_contents('php://input'));
} catch (SignatureException $e) {
    http_response_code(400);
    return;
}

$eventKey = $payload->getUuid() . ':' . $payload->getStatus();

// Atomic "claim" — return false if the key already exists.
// SQL equivalent: INSERT ... ON CONFLICT DO NOTHING; check affected rows.
if (!$db->claimWebhookEvent($eventKey)) {
    http_response_code(200);
    echo 'OK (duplicate)';
    return;
}

// First time we've seen this event — do the work.
try {
    processWebhook($payload);
} catch (\Throwable $e) {
    // Release the claim so Heleket's retry can succeed.
    $db->releaseWebhookEvent($eventKey);
    http_response_code(500);
    throw $e;
}

http_response_code(200);
echo 'OK';
```

### Pattern with a real database

In MySQL/PostgreSQL, use a table with a unique key on the event identifier and rely on `INSERT ... ON DUPLICATE KEY UPDATE` (MySQL) or `ON CONFLICT DO NOTHING` (Postgres) to atomically claim the event:

```sql
CREATE TABLE heleket_webhook_events (
    event_key   VARCHAR(128) PRIMARY KEY,
    payload     JSON NOT NULL,
    received_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    processed_at TIMESTAMP NULL
);
```

```php
// MySQL claim — returns true if this row was actually inserted.
$stmt = $pdo->prepare(
    'INSERT IGNORE INTO heleket_webhook_events (event_key, payload) VALUES (?, ?)'
);
$stmt->execute([$eventKey, json_encode($payload->toArray())]);
$claimed = $stmt->rowCount() === 1;

if (!$claimed) {
    http_response_code(200);
    return;
}

processWebhook($payload);
$pdo->prepare('UPDATE heleket_webhook_events SET processed_at = NOW() WHERE event_key = ?')
    ->execute([$eventKey]);
```

### Pattern with Redis (lighter, but TTL-bound)

```php
// SET key value NX EX 86400 — set only if not exists, expire after 24h.
$claimed = $redis->set("heleket:webhook:{$eventKey}", '1', ['NX', 'EX' => 86400]);
if (!$claimed) {
    http_response_code(200);
    return;
}
processWebhook($payload);
```

The Redis variant is simpler but forgets after the TTL — pick a window longer than Heleket's retry window (24h is a safe default).

### Why `(uuid, status)`, not just `uuid`

A single invoice can legitimately progress through several final statuses — for example `check` → `paid`, or `wrong_amount_waiting` → `paid`. Deduping by `uuid` alone would drop the second event and your accounting would diverge. The `(uuid, status)` pair is unique per real state transition.

## IP allowlisting

Heleket sends webhooks from `31.133.220.8`. If your reverse proxy can filter by source IP, allow only that one. **Don't rely on IP alone** — signature verification is the authoritative trust boundary.

## Local development

Use a tunnel to expose your local machine:

```bash
# Run the bundled handler on :8000
make example-webhook

# In another terminal, start ngrok
ngrok http 8000
```

Then set the resulting URL as `url_callback` when creating an invoice, or call `testWebhook()`:

```php
$payment->testWebhook(
    type:    'payment',
    url:     'https://abc123.ngrok.io',
    currency:'USD',
    network: 'tron',
    status:  'paid',
    orderId: 'order-42'
);
```

## When signatures fail — debug recipe

Capture the failing payload, then:

```bash
echo '<paste the JSON here>' | bin/heleket-webhook-inspect --key=$HELEKET_PAYMENT_KEY
```

See [07 — Debugging](07-debugging.md) for the inspector CLI in detail.
